B03-DN / Thông tư 200 — Cách tính giá trị chỉ tiêu (tài liệu kỹ thuật)

Phương pháp trực tiếp · Module l10n_vn_b03dn_direct_report · Đối tượng: developer / triển khai

1. Phạm vi

Báo cáo B03-DN — Lưu chuyển tiền tệ, phương pháp trực tiếp theo hướng dẫn Thông tư 200/2014 (TT200). Số liệu không hard-code trong Python: mỗi chỉ tiêu là một dòng cấu hình l10n.vn.b03dn.line trong template l10n.vn.b03dn.template.

Engine tính toán: model abstract l10n.vn.b03dn.engine, method compute_period(template, company, date_from, date_to) → lines_by_code[code] = { amount, aml_ids }.

2. Luồng tổng thể (đầu vào → đầu ra)

INPUT
  template.line_ids (sorted sequence, id)
  company, date_from, date_to

BƯỚC A — Tập TK tiền
  _cash_account_ids: cash_account_ids HOẶC (111%, 112%, 113% + b03dn_cash_equiv_account_ids)

BƯỚC B — Dòng tiền trong kỳ
  account.move.line: account_id ∈ cash, date ∈ [from,to], posted

BƯỚC C — Fragment
  Mỗi dòng tiền + các dòng cùng move → 1..N fragment:
    cash_aml_id, amount (>0), is_inflow, counterpart_code, counterpart_account_id

BƯỚC D — Gán leaf (first match)
  Với mỗi fragment, duyệt leaf theo sequence:
    chọn pattern theo is_inflow → so khớp đối ứng → tag include/exclude → extra_domain
    signed = amount nếu inflow else -amount; × amount_multiplier
    buckets[code] += signed ; break

BƯỚC E — opening_cash, fx_adjustment (không qua fragment)

BƯỚC F — aggregate (sum_expression, lặp đến ổn định)

OUTPUT
  { lines_by_code, account_ids }

3. Xác định tiền vào / ra (is_inflow)

Trong _fragment_from_cash_line (trên một account.move.line là TK tiền):

Điều kiện (sau làm tròn tiền tệ)is_inflowĐối ứng lấy từ
Nợ tiền > 0, Có tiền ≈ 0TrueCác dòng khác trên move có Có > 0 → credit_pairs()
Có tiền > 0, Nợ tiền ≈ 0FalseCác dòng khác có Nợ > 0 → debit_pairs()
Khác (vd cả Nợ và Có tiền)—Không tạo fragment (return [])

Chứng từ chỉ quyết giữa các TK tiền → bỏ qua (không fragment).

Nhiều đối ứng cùng phía → _allocate_split_amounts chia amount cho từng fragment.

4. credit_account_patterns và debit_account_patterns

Hai trường Char, danh sách mẫu mã TK phân tách dấu phẩy. Quy ước: hậu tố % = prefix; không có % = khớp đúng mã.

Sau khi đã biết is_inflow của fragment, hàm _leaf_rule_counterpart_pattern_source(rule, is_inflow):

  • is_inflow True → dùng chỉ rule.credit_account_patterns (đối ứng phía Có, khớp cột «TK Có» trên bảng TT200 cho luồng thu tiền).
  • is_inflow False → dùng chỉ rule.debit_account_patterns (đối ứng phía Nợ, «TK Nợ» cho luồng chi).

Với mỗi leaf đang xét, parse pattern của chính leaf đó, so với frag["counterpart_code"] (có thể tách nhiều mã nếu chuỗi có dấu phẩy) qua _code_matches_patterns.

Rule leaf bị bỏ qua nếu không có pattern (sau parse) và không có tag_ids — cho phép skip leaf “trống”.

5. Thẻ và domain

  • exclude_tag_ids: nếu bất kỳ thẻ loại trừ nào có trong “pool” (theo tag_source giống thẻ yêu cầu) → không khớp leaf này.
  • tag_ids + tag_match_mode (all/any): pool từ tag_source (counterpart_account / cash_line / either).
  • extra_domain: domain Odoo trên dòng tiền, safe_eval, áp sau khi khớp pattern.

6. Ưu tiên leaf

leaf_lines_ord = leaf sort theo (sequence, id). Với mỗi fragment, duyệt leaf theo thứ tự; leaf đầu tiên thỏa đủ điều kiện nhận fragment → break. Các leaf sau không được xét cho fragment đó.

7. line_kind (cách có số cho “chỉ tiêu”)

line_kindNguồn sốGhi chú
none0Tiêu đề / trống
leafCộng dồn signed từ fragment khớpaml_ids = id các dòng tiền

8. Liên hệ TT200

Bảng Thông tư (Chỉ tiêu / Mã số / TK Nợ / TK Có) được phản ánh trong cấu hình template dữ liệu data/b03dn_template_tt200_direct.xml: tên chỉ tiêu, code, debit_account_patterns, credit_account_patterns, loại dòng (leaf / tổng hợp / đầu kỳ / tỷ giá).

Cột TK tiền (111, 112…) trên TT200 thường chỉ để đối chiếu; engine quét TK tiền theo _cash_account_ids, không lọc bằng hai pattern trên cho “cột tiền”.

9. Ví dụ minh họa — data vào / data ra

Đơn vị: VND. Giả sử chỉ quét TK tiền 1111. Template rút gọn hai leaf: sequence 10 → mã 01 (credit_account_patterns = 511%,131%); sequence 20 → mã 02 (debit_account_patterns = 331%). amount_multiplier = 1, không thẻ, không extra_domain.

9.1. Thu tiền bán hàng (tiền vào)

Data vào — các dòng account.move.line (cùng chứng từ A, đã post, trong kỳ):

idMoveTKNợCó
100A111120.000.0000
101A5111020.000.000

Bước quét tiền: vào cash_amls chỉ dòng 100 (TK 1111).

Data trung gian — fragment:

cash_aml_id=100, amount=20_000_000, is_inflow=True,
counterpart_code="5111", counterpart_account_id=<id 5111>

Gán leaf: is_inflow → dùng credit_account_patterns của từng leaf; mã 01 khớp tiền tố 511%. signed = +20.000.000.

Data ra — phần lines_by_code (trích):

"01": { amount: 20_000_000, aml_ids: [100] }

9.2. Chi trả nhà cung cấp (tiền ra)

Data vào — chứng từ B:

idMoveTKNợCó
200B331110.000.0000
201B1111010.000.000

cash_amls: dòng 201.

Fragment:

cash_aml_id=201, amount=10_000_000, is_inflow=False,
counterpart_code="3311"

Gán leaf: dùng debit_account_patterns → mã 02 khớp 331%. signed = -10.000.000.

Data ra:

"02": { amount: -10_000_000, aml_ids: [201] }

9.3. Một lần xuất quỹ, hai đối ứng Nợ (phân bổ)

Data vào — chứng từ C (trả NCC + VAT cùng một dòng Có tiền):

idMoveTKNợCó
301C331120.000.0000
302C33313.600.0000
303C1111023.600.000

cash_amls: dòng 303 (Có 23.600.000).

Data trung gian — hai fragment (trọng số Nợ 20M và 3,6M):

{ cash_aml_id:303, amount≈20_000_000, is_inflow:False, counterpart_code:"3311" }
{ cash_aml_id:303, amount≈3_600_000,  is_inflow:False, counterpart_code:"3331" }

Giả sử còn leaf mã 05 với debit_account_patterns = 333% (hoặc 3331 chính xác) và sequence sao cho thử sau mã 02: fragment 3311 khớp 02 trước; fragment 3331 không khớp 331% → thử tiếp → khớp 05. Data ra kiểu: "02": -20.000.000, "05": -3.600.000 (cùng aml_ids chứa 303 nếu cả hai phần đều từ dòng tiền đó).

Drill-down aml_ids: với leaf, tập id trỏ tới dòng tiền (cash_aml_id), không nhất thiết là dòng doanh thu hay phải trả.

9.4. Bộ chứng từ mẫu — 27 bút toán (đủ mọi leaf 01–07, 21–27, 31–36 + nội bộ tiền)

Căn cứ data/b03dn_template_tt200_direct.xml. Số tiền VND minh họa — có thể thay đổi tùy ý.

9.5. Báo cáo (đầu ra) từ đúng bộ B01–B26 — lines_by_code và aggregate

Giả định kỳ báo cáo chỉ chứa các chứng từ tiền tương ứng 9.4.1 (và không có phát sinh tiền khác). Khi đó compute_period cho phần leaf có dạng (trích — aml_ids bỏ qua ở đây):

// Leaf (VND) — dấu: inflow +, outflow −
"01":  93_400_000    // B01+B02+B03+B04+B06
"02": -45_500_000    // B07+B08+B14(20M)
"03": -22_000_000    // B09
"04":  -1_100_000    // B10
"05":  -4_000_000    // B11
"06":   1_200_000    // B05
"07": -16_500_000    // B12+B13+B14(3,6M)
"21": -45_000_000    // B15
"22":   6_000_000    // B16
"23": -10_000_000    // B17
"24":  10_000_000    // B18
"25": -15_000_000    // B19
"26":  15_000_000    // B20
"27":           0    // không có fragment chỉ khớp 27 trong bộ mẫu
"31":  50_000_000    // B21
"32":  -5_000_000    // B22
"33": 100_000_000    // B23
"34": -40_000_000    // B24
"35":  -3_000_000    // B25
"36":  -8_000_000    // B26

Sau bước aggregate (theo sum_expression trong XML): 20 = 01+02+03+04+05+06+07 → 5_500_000; 30 = 21+…+27 → -39_000_000; 40 = 31+…+36 → 94_000_000; 50 = 20+30+40 → 60_500_000. Bút B27 không thay đổi bất kỳ bucket leaf nào. 60 (đầu kỳ), 61 (FX), 70 (cuối kỳ) cần thêm dữ liệu ngoài bộ chứng từ mẫu.
